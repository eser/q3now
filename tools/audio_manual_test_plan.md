# Audio Manual Test Plan — q3now miniaudio backend

## Purpose

Manual gameplay test plan for the new miniaudio audio backend
(`code/client/snd_miniaudio.c`) that replaced DirectSound / ALSA /
OSS / SDL2. Satisfies acceptance criteria ac-1 through ac-7 and
ac-29 of the `replace-openal-miniaudio-q3now-audio-backend` spec.

This file is the **plan**. Execution is a separate activity performed
by a human at a real machine. Do not edit this file with results —
copy it and fill the copy in.

---

## How to use this plan

1. Build the engine fresh (`make` from the q3now project root).
2. Make a working copy:
   `cp tools/audio_manual_test_plan.md tools/audio_manual_test_results_YYYYMMDD.md`
   (use today's date).
3. Walk every test in order. For each one: perform the **Setup**,
   take the **Action**, compare against **Expected**, tick `[ ]`,
   and replace the `Result:` line with PASS / FAIL / N/A plus a
   one-line note.
4. When finished, paste captured console output at the bottom of
   the results file and commit it alongside the spec close-out.
5. Failures: do not skip remaining tests. Mark clearly and capture
   any console output.

---

## Pre-test setup

Build, then launch from a Bash shell so console output is captured:

```
make
make run-game DEV=1 2>&1 | tee /tmp/audio_manual_test_console.log
```

The `run-game` target rebuilds the macOS `.app` bundle as needed.
Stderr (which carries the console output) is mirrored to the log
file. Leave the terminal visible — several tests refer back to it.

You will need: q3dm6 installed, a map with OGG music (or the menu
cinematic), wired or USB headphones you can physically unplug, and
the in-game console key (default `~`).

---

## Console output to capture

Watch the launching terminal and copy the following into the results
file:

1. The startup line beginning with `miniaudio:` (device name, sample
   rate, channel count, period size).
2. Any line containing `underrun`.
3. Any audio-subsystem error line (prefixes `s_`, `snd`, `miniaudio`).
4. Any crash, assertion, or stack trace.
5. The output of `s_info` and `sndlist` from section 4.

If nothing matches, write "no audio errors observed" in the captured
section.

## 1. Smoke tests

These verify the engine is alive and the audio path is producing sound
at all. If any smoke test fails, stop and investigate before continuing.

### 1.1 Engine startup banner

- Setup: Launch the game per the Pre-test setup section.
- Action: Watch the terminal as the engine boots. Do not press any
  keys.
- Expected: A single line of the form
  `miniaudio: <device-name>, <hz> Hz, <channels> ch, <frames>-frame periods`
  appears during startup, before the main menu draws. The exact format
  is whatever `snd_miniaudio.c` prints — the key requirement is that
  the line exists, names a real device, and reports a non-zero sample
  rate.

## 2. Spatial audio tests

These verify that the existing `S_SpatializeOrigin` panning and
distance attenuation paths still work through the new backend.

### 2.1 Weapon firing on q3dm6

- Setup: Console: `map q3dm6`, then `give all`.
- Action: Cycle through every weapon and fire each at least once
  while standing still in the spawn room.
- Expected: Each weapon plays its correct firing sound. The sound
  appears to come from your own position (centered, no obvious
  panning bias), with the correct timbre per weapon.
- [ ] Checked
- Result: PASS / FAIL / N/A

### 2.2 Walk around a sound source — panning

- Setup: On q3dm6, locate a continuous sound source (the teleporter
  hum, a jumppad, or any ambient loop). Stand a few meters away.
- Action: Strafe a full circle around the source while keeping your
  view aimed at it. Then turn your view through a full 360 while
  standing still next to it.
- Expected: The panning shifts smoothly between left and right as
  the source moves across your view. No abrupt jumps, no silence
  zones, no stuck channel.
- [ ] Checked
- Result: PASS / FAIL / N/A

### 2.3 Distance attenuation

- Setup: Same continuous sound source as 2.2.
- Action: Walk in a straight line away from the source until it is
  inaudible. Then walk back.
- Expected: Volume fades smoothly with distance. No sudden cutoffs,
  no constant volume, no inverse behavior. Returning closer brings
  the volume back smoothly.
- [ ] Checked
- Result: PASS / FAIL / N/A

- Expected: After the restart, `s_info` reports the device you set

  falls back to the default and prints a warning rather than

- Action: Console: `s_latency 12`, then `snd_restart`. Then run

  changed in the direction implied by the higher latency value.
